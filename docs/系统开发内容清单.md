# 基于区块链的学术会议匿名审稿与版权存证系统——开发内容清单

## 1. 项目目标与范围

围绕“学术会议匿名审稿 + 版权存证”构建一套可运行系统，达成以下目标：
- 解决传统审稿匿名性不足、评审结果可篡改、版权归属难追溯的问题。
- 实现稿件从上传、存证、分配审稿、匿名评审、结果裁定到查询追溯的全流程闭环。
- 基于联盟链（FISCO BCOS）提供不可篡改证据链，支撑版权确权与审计。

系统边界：
- 业务数据（用户、稿件元数据、任务、配置、日志）存储在 MySQL。
- 关键证据数据（哈希、评审记录摘要、裁定结果、交易信息）上链存储。
- 前后端分离架构，后端与链通过 Web3/WeBASE 交互。

## 2. 技术栈与工程基础（必须完成）

## 后端
- Node.js（Express 或 NestJS）+ TypeScript。
- 实现统一鉴权（JWT）与 RBAC 权限控制。
- 封装链交互服务（合约调用、交易回执、查询、异常重试）。

## 前端
- Vue 3 + TypeScript + Element Plus。
- 按角色（管理员/作者/审稿人）提供差异化门户与菜单。
- 关键链上操作必须具备 loading、成功/失败提示、TxHash 展示。

## 区块链
- FISCO BCOS 联盟链网络搭建与接入（建议 >=4 节点满足 PBFT 容错要求）。
- Solidity 智能合约开发、部署、版本管理。
- 通过 WeBASE 或 RPC 接口完成合约调用与链数据查询。

## 运行与部署
- Docker 化部署（前端 Nginx、后端服务、数据库、链节点）。
- 环境分离（开发/测试/演示）。
- 基础日志、健康检查、错误告警能力。

## 3. 角色与权限体系（必须完成）

- **管理员**：用户审核与角色分配、审稿任务分配、链节点监控、交易溯源、合约管理、会议参数配置、权限映射。
- **作者**：注册登录、稿件上传、触发/查看哈希存证、版权查询、查看评审结果。
- **审稿人**：匿名身份登录、查看任务、下载脱敏稿件、提交加密评审意见与评分。

最小权限要求：
- 作者仅访问本人稿件与存证/评审结果。
- 审稿人仅访问被分配且脱敏后的稿件与本人评审记录。
- 管理员拥有全局管理能力并记录关键操作日志。

## 4. 功能模块开发清单（核心）

## 4.1 用户管理子系统（高优先级）
- 实名注册：字段校验（邮箱唯一、密码强度等）。
- 登录认证：JWT 会话管理。
- 钱包生成：为用户生成链上地址、公钥/私钥；私钥必须加密存储。
- 角色分配：管理员审核并分配“作者/审稿人”等角色。

## 4.2 版权存证子系统（高优先级）
- 稿件上传：仅支持 PDF（<=20MB），保存文件与元数据。
- 哈希存证：计算 SHA-256，提交 `fileHash + authorAddress + timestamp` 上链。
- 存证状态管理：保存 transaction hash、block height、存证时间，回写业务库。
- 版权查询：输入哈希或上传文件比对，返回链上确权结果。

## 4.3 匿名审稿子系统（高优先级）
- 任务分配：管理员手动/自动分配，每稿默认 3 名审稿人。
- 自动匹配：按关键词/研究方向 + 当前负载进行推荐。
- 匿名评审：审稿人提交评分、评语、推荐意见；支持签名与链上存证。
- 结果裁定：根据预设规则（阈值、加权/均值策略）自动给出录用结论并上链。

## 4.4 区块链管理子系统（中优先级）
- 节点监控：区块高度、节点状态、TPS、延迟、PBFT 状态等可视化。
- 交易溯源：按 TxHash 查询并解析交易详情、事件日志、关联合约调用。
- 合约管理：部署、版本管理、ACL 控制、紧急冻结/切换（如采用代理模式）。

## 4.5 系统配置子系统（中优先级）
- 参数设定：投稿起止时间、审稿周期、评分项与权重、录用阈值等。
- 权限映射：角色-菜单-操作级别（增删改查）可视化配置。
- 配置生效：参数更新后影响投稿开关、分配逻辑与裁定计算。

## 5. 智能合约开发清单（必须完成）

建议至少实现以下合约能力：
- 版权存证合约：登记稿件哈希、作者地址、时间戳；支持哈希查询。
- 审稿任务合约：登记分配关系、任务状态、审稿提交状态。
- 评审记录合约：保存评审摘要/哈希、评分结果与签名校验信息。
- 裁定结果合约：保存最终分数与录用状态，提供可验证查询接口。

合约工程要求：
- 明确事件（Event）设计，便于后端订阅与审计。
- 关键函数访问控制（仅管理员/系统地址可调用）。
- 升级策略（代理或版本迁移）与回滚预案。

## 6. 数据模型与存储清单（必须完成）

数据库核心表（至少）：
- `UserInfo`：用户实名信息、角色、状态、钱包地址关联。
- `PaperInfo/PaperRepo`：稿件元数据、文件路径、哈希、存证状态。
- `ReviewTask`：稿件-审稿人映射、截止时间、状态、链上关联哈希。
- `ReviewResult`：评分、评语（加密/摘要）、推荐意见、交易哈希。
- `ConferenceConfig`：会议参数与裁定规则。
- `ChainTransaction`：链上交易流水缓存与索引。
- `SystemLog`：操作日志、安全日志、异常日志。

数据流要求：
- 高频查询和业务状态走 MySQL。
- 可追溯证据与不可篡改凭据走链上。
- 通过交易哈希建立“业务记录 <-> 链上记录”双向索引。

## 7. 非功能与安全要求（必须完成）

## 性能
- 支持 >=100 并发用户访问。
- 支持 >=100 TPS 级别链交互目标（按环境评估达标）。
- 常规页面查询/跳转响应 <=2 秒。
- 链上确认时间（正常网络）<=5 秒。
- 10MB 稿件上传+哈希计算 <=3 秒。

## 安全
- 密码使用 BCrypt 哈希存储。
- 全站 HTTPS，防中间人窃听。
- 防 SQL 注入、XSS、CSRF。
- 私钥安全存储（加密、权限隔离、最小暴露）。
- 审稿匿名隔离（作者信息脱敏展示，审稿端不可见真实身份）。

## 稳定与可扩展
- 支持 7x24 运行目标，具备节点故障切换与重试机制。
- 模块化设计，后续可扩展新业务模块。
- 支持联盟链节点横向扩展与合约平滑升级。

## 8. 界面与报表开发清单（必须完成）

界面要求：
- 统一风格与组件规范，支持面包屑与状态标签。
- 表单即时校验（上传类型、大小、评分范围等）。
- 列表页支持分页、筛选、排序。
- 审稿界面对作者信息进行系统化脱敏。

报表/证书要求：
- 自动生成《区块链存证证书》：含稿件信息、SHA-256、TxHash、区块高度、时间戳。
- 提供二维码，跳转到交易查询页面进行真伪验证。
- 生成《稿件评审结果通知》：最终裁定、评分摘要、后续指引。

## 9. 测试与验收清单（必须完成）

测试类型：
- 功能测试：覆盖五大子系统核心流程。
- 安全测试：鉴权、越权、注入、XSS/CSRF、私钥保护。
- 性能测试：并发访问、链上提交时延、上传处理时延。
- 稳定性测试：节点异常、链交互失败重试、长时间运行。

关键验收场景（建议最少）：
- 作者注册 -> 上传稿件 -> 存证成功 -> 查询可验证。
- 管理员分配审稿 -> 审稿人匿名提交 -> 系统自动裁定 -> 作者可查看结果。
- 管理员可通过 TxHash 溯源任一关键操作。
- 系统参数修改后，新流程规则即时生效。

论文与答辩交付：

## 11. 开发优先级建议（实施顺序）

1) 基础工程与鉴权：用户、登录、角色权限、项目骨架  
2) 稿件上传与哈希存证：先打通“上传-上链-查询”闭环  
3) 匿名审稿主流程：任务分配、匿名评审、结果裁定  
4) 区块链运维能力：节点监控、交易溯源、合约管理  
5) 系统配置与报表证书：参数、权限映射、证书导出  
6) 性能安全强化与验收测试：压测、安全加固、稳定性验证
