1 引言

1.1 编制目的
本报告用于定义“基于区块链的学术会议匿名审稿与版权存证系统（ConfChain）”的功能需求、非功能需求和运行约束。
主要目的如下：
1) 为开发人员提供明确、可实现、可测试的需求基线；
2) 为测试人员编写测试用例提供依据；
3) 为后续概要设计、详细设计、系统验收和项目交接提供依据。

1.2 范围
1) 产品标识
- 中文名称：基于区块链的学术会议匿名审稿与版权存证系统
- 系统简称：ConfChain

2) 产品功能范围（当前实现）
- 账号注册登录、JWT 鉴权、角色权限控制（ADMIN/AUTHOR/REVIEWER）；
- 投稿上传、文件哈希计算、版权存证、公开验证；
- 审稿任务分配（手动/自动）、匿名评审提交、稿件裁定；
- 区块链节点状态查看、交易溯源、交易统计与业务追踪；
- 裁定阈值配置（acceptThreshold）。

3) 目标
- 通过链上不可篡改凭证提升版权证明能力；
- 通过流程化审稿提高评审过程规范性与可追踪性；
- 实现论文投稿、审稿、裁定、验证的闭环管理。

1.3 预期读者和阅读建议

2 项目概述

2.1 目标
系统用于支撑学术会议场景下的投稿存证与匿名审稿管理，解决传统流程中：
- 投稿时间与版权证明留痕不足；
- 审稿流转过程可追踪性不足；
- 裁定依据与过程留档不统一。

2.2 范围
系统覆盖以下业务范围：
- 用户层：注册、登录、角色管理；
- 投稿层：稿件上传、哈希计算、版权存证、验证查询；
- 审稿层：任务分配、评审提交、结果裁定；
- 管理层：节点状态、交易追踪、系统阈值配置。

2.3 用户特点
1) 作者（AUTHOR）
- 提交稿件并查看自身稿件状态；
- 可查询和验证版权存证信息。

2) 审稿人（REVIEWER）
- 接收审稿任务并提交评分、建议与评语；
- 可查看被分配稿件的脱敏信息并下载稿件。

3) 管理员（ADMIN）
- 维护用户角色；
- 管理审稿分配和裁定；
- 查看区块链状态与交易流水；
- 设置系统裁定阈值。

2.4 假定条件和约束限制
- 假定 WeBASE-Front 与 FISCO BCOS 节点可访问；
- 当链路不可用时，系统允许模拟回执（simulated=true）保证流程可继续；
- 文件上传大小不超过 20MB；
- 系统使用固定角色模型，不支持动态自定义角色；
- 当前仅提供“裁定阈值”配置，不提供完整权限矩阵配置页面。

2.5 运行环境

2.5.1 硬件环境
服务器建议配置：
- CPU：4核及以上；
- 内存：16GB 及以上；
- 磁盘：SSD 500GB 及以上。

客户端建议配置：
- CPU：双核及以上；
- 内存：4GB 及以上（建议 8GB）；
- 分辨率：1920x1080。

2.5.2 软件环境
- 服务端：Node.js 18+，NestJS 11，TypeScript；
- 客户端：Vue 3 + Vite + Element Plus；
- 数据库：MySQL 8.0（Prisma ORM）；
- 区块链：FISCO BCOS 3.x + WeBASE-Front；
- 合约：Solidity 0.8.11；
- 协议：HTTP/HTTPS，JSON-RPC。


3 业务分析

3.1 组织机构结构分析
- 会议管理员：负责账号角色、审稿分配、裁定、系统参数配置；
- 审稿专家：负责对分配稿件进行评审提交；
- 作者：负责投稿、查看稿件进度、验证版权信息。

3.2 业务流程分析
1) 注册登录流程
用户注册后写入 User 表，默认角色为 AUTHOR，可由管理员调整角色。

2) 投稿存证流程
作者上传稿件，系统计算 fileHash 和 metadataHash，写入 Paper，调用链服务存证并写入 ChainTransaction。

3) 匿名审稿流程
管理员分配 ReviewTask；审稿人提交 ReviewResult；系统记录审稿上链回执。

4) 裁定流程
管理员触发裁定，系统读取 ReviewResult 计算平均分并基于阈值判定 ACCEPTED/REVISION/REJECTED，再写入链交易流水。

5) 公开验证流程
任意用户可通过哈希或上传文件验证版权存证记录（无需登录）。


4 用户需求

4.1 总体功能

4.1.1 系统功能框图说明
系统由五个子系统组成：
- SYS1 用户管理
- SYS2 版权存证
- SYS3 匿名审稿
- SYS4 区块链管理
- SYS5 系统配置

4.1.2 总体功能分类描述
表4-1 总体功能分类
- SYS1 用户管理：注册登录、个人信息与角色管理；
- SYS2 版权存证：上传投稿、链上存证、公开验证；
- SYS3 匿名审稿：分配任务、提交评审、执行裁定；
- SYS4 区块链管理：节点状态、交易查询、统计展示；
- SYS5 系统配置：裁定阈值配置。

4.2 用户管理子系统
1) 注册（SYS1-01）：用户提供姓名、邮箱、密码完成注册；
2) 登录（SYS1-02）：账号密码登录并获取 JWT；
3) 角色分配（SYS1-03）：管理员修改用户角色。

说明：
- 当前实现中“钱包地址、公私钥”由后端随机生成并存储于 User 表字段；
- 非通过链节点实时生成钱包服务。

4.3 版权存证子系统
1) 投稿并存证（SYS2-01）
- 接收标题、摘要、关键词和文件；
- 计算 SHA-256 文件哈希并调用链服务；
- 返回 txHash、blockHeight、certifiedAt。

2) 版权验证（SYS2-02）
- 支持 fileHash 查询；
- 支持上传文件计算哈希后查询。

3) 稿件管理（SYS2-03）
- 作者查询我的稿件；
- 作者在 REVISION 状态下可提交修订稿再次存证；
- 作者/管理员可下载稿件，审稿人可下载已分配稿件。

4.4 匿名审稿子系统
1) 手动分配（SYS3-01）
管理员为指定稿件分配审稿人和截止日期。

2) 自动分配（SYS3-02）
系统按“当前任务量最少优先”选择审稿人，不以关键词匹配为主规则。

3) 评审提交（SYS3-03）
审稿人提交 score/recommendation/comment；系统将 comment 的哈希写链并保存业务结果。

4) 结果裁定（SYS3-04）
管理员手动触发裁定（非自动触发）；依据平均分和阈值得出最终状态。

4.5 区块链管理子系统
1) 节点状态（SYS4-01）：查看 connected、nodeCount、blockNumber、chainId；
2) 交易溯源（SYS4-02）：按 txHash 查询链上详情；
3) 交易统计（SYS4-03）：统计存证、审稿、裁定交易数量；
4) 业务追踪（SYS4-04）：按 bizId 查看关联流水；
5) 合约信息查看（SYS4-05）：查看合约地址与函数信息。

说明：
- 当前实现无“在线部署/升级/冻结合约”操作页面。

4.6 系统配置子系统
1) 裁定阈值配置（SYS5-01）
管理员设置 acceptThreshold，影响裁定分界线。

说明：
- 当前实现未提供“权限映射矩阵”配置功能；
- 评审权重、投稿周期等参数在后端有默认值，但前台仅开放阈值配置。


5 数据描述

5.1 数据流程说明
- 用户请求进入 API 层；
- 高频业务数据写入 MySQL；
- 关键证明数据经 WeBASE 调用上链；
- 链回执落库到业务表与 ChainTransaction 表，支持后续查询与审计。

5.2 数据字典
表5-1 核心数据表
1) User
- 用户基础信息、登录凭证摘要、角色、钱包地址、公私钥。

2) Paper
- 稿件元数据、文件路径、文件哈希、状态、链回执信息。

3) ReviewTask
- 稿件与审稿人分配关系、截止时间、任务状态。

4) ReviewResult
- 评分、录用建议、评语原文、评语哈希、交易哈希。

5) ConferenceConfig
- 会议配置（当前主要使用 acceptThreshold）。

6) ChainTransaction
- 业务链交易流水（COPYRIGHT_CERTIFY/REVIEW_SUBMIT/ADJUDICATE）。

说明：
- 当前实现中不存在 SystemLog、WalletEnti、PermissionEnti 等独立业务表。


6 功能需求

6.1 功能需求总述
系统按当前实现划分为 5 个子系统，对应 15 项核心需求。

6.1.1 功能需求总表
表6-1 功能需求表
- SRS_SYS1_01 注册
- SRS_SYS1_02 登录
- SRS_SYS1_03 角色分配
- SRS_SYS2_01 投稿并存证
- SRS_SYS2_02 版权验证（哈希）
- SRS_SYS2_03 版权验证（文件上传）
- SRS_SYS2_04 我的稿件/下载/修订
- SRS_SYS3_01 手动分配
- SRS_SYS3_02 自动分配
- SRS_SYS3_03 提交评审
- SRS_SYS3_04 执行裁定
- SRS_SYS4_01 节点状态
- SRS_SYS4_02 交易溯源与统计
- SRS_SYS4_03 业务流水查询
- SRS_SYS5_01 裁定阈值配置

优先级说明：
- 高：SYS1/SYS2/SYS3 核心流程需求；
- 中：SYS4/SYS5 运维和参数需求。

6.1.2 角色与权限需求
表6-2 角色权限分配
- ADMIN：全量管理权限（用户、审稿、区块链、配置）；
- AUTHOR：投稿、验证、查看本人稿件与裁定结果；
- REVIEWER：查看本人任务、提交评审、下载已分配稿件。

6.2 用户管理功能需求
6.2.1 注册（SRS_SYS1_01）
输入：name、email、password、role(可选)
处理：校验唯一性和密码规则，默认角色 AUTHOR
输出：用户创建结果

6.2.2 登录（SRS_SYS1_02）
输入：email、password
处理：校验凭证
输出：accessToken + 用户信息

6.2.3 角色分配（SRS_SYS1_03）
输入：用户ID + 新角色
处理：管理员调用更新角色接口
输出：更新后的用户信息

6.3 版权存证功能需求
6.3.1 投稿并存证（SRS_SYS2_01）
输入：title、abstract、keywords、file
处理：上传文件、计算 hash、写库、调用链、回写流水
输出：paper 记录 + txHash/blockHeight

6.3.2 版权验证（SRS_SYS2_02/SYS2_03）
输入：fileHash 或 file
处理：计算/查询哈希并检索存证记录
输出：found=true/false 及存证详情

6.3.3 下载与修订（SRS_SYS2_04）
输入：paperId + 身份
处理：权限校验、下载文件；REVISION 状态允许修订并重新存证
输出：下载流或修订结果

6.4 匿名审稿功能需求
6.4.1 手动分配（SRS_SYS3_01）
输入：paperId、reviewerIds、deadlineAt
输出：ReviewTask 列表，稿件状态变为 UNDER_REVIEW

6.4.2 自动分配（SRS_SYS3_02）
输入：paperId、count、deadlineAt
处理：根据审稿任务负载排序分配
输出：ReviewTask 列表

6.4.3 提交评审（SRS_SYS3_03）
输入：taskId、score(0-100)、recommendation、comment
处理：写审稿结果、更新任务状态、记录链回执
输出：评审提交成功信息

6.4.4 裁定（SRS_SYS3_04）
输入：paperId
处理：计算平均分 + 阈值规则，更新 Paper 状态并写链
输出：finalStatus + averageScore + txHash

6.5 区块链管理功能需求
6.5.1 节点状态（SRS_SYS4_01）
输入：无
输出：connected、nodeCount、blockNumber、chainId

6.5.2 交易溯源与统计（SRS_SYS4_02）
输入：txHash 或分页参数
输出：交易详情、分类统计、交易列表

6.5.3 业务流水查询（SRS_SYS4_03）
输入：bizId
输出：该业务关联的链流水列表

6.6 系统配置功能需求
6.6.1 裁定阈值配置（SRS_SYS5_01）
输入：acceptThreshold
处理：保存到 ConferenceConfig
输出：配置更新成功


7 系统分析与用例

7.1 系统角色分析
- ADMIN：系统治理者；
- AUTHOR：投稿与版权主体；
- REVIEWER：匿名评审执行者。

7.2 系统用例分析
关键用例：
- UC01 用户注册与登录
- UC02 投稿并存证
- UC03 公开版权验证
- UC04 审稿任务分配
- UC05 审稿提交
- UC06 稿件裁定
- UC07 区块链交易追踪
- UC08 阈值配置

7.3 关键用例描述
用例A：投稿并存证
- 参与者：AUTHOR
- 前置条件：已登录
- 主流程：上传稿件 -> 计算哈希 -> 上链 -> 返回存证结果
- 后置条件：Paper.status=CERTIFIED，生成 ChainTransaction

用例B：裁定
- 参与者：ADMIN
- 前置条件：稿件已有审稿结果
- 主流程：读取评分 -> 计算平均分 -> 阈值判定 -> 上链
- 后置条件：Paper.status 更新为 ACCEPTED/REVISION/REJECTED

7.4 系统概念模型
- 视图模型：按角色菜单展示不同页面；
- 逻辑模型：Controller -> Service -> Prisma + BlockchainService；
- 实体模型：User、Paper、ReviewTask、ReviewResult、ConferenceConfig、ChainTransaction。


8 非功能需求

8.1 性能需求
- 普通页面响应时间目标：2秒内；
- 10MB 以内文件上传与哈希计算目标：3秒内（网络正常）；
- 链路调用在节点正常时应有可接受响应，异常时允许模拟降级。

8.2 安全保密需求
- 身份认证：JWT；
- 密码存储：BCrypt 哈希；
- 权限控制：RBAC（JwtAuthGuard + RolesGuard）；
- 输入防护：ValidationPipe 白名单校验；
- 基础防护：Helmet 安全头、CORS 控制。

说明：
- 当前实现中审稿评语明文保存在业务库，链上存储其哈希摘要；
- 不宣称已实现“全量评语密文上链可解密”能力。

8.3 扩展性需求
- 可扩展更多会议配置项（评审权重、时间窗口等）；
- 可扩展可视化权限映射；
- 可扩展链交易补偿与重试机制。

8.4 稳定性需求
- 区块链不可用时允许模拟模式继续业务流程；
- 核心异常需记录日志并提示用户可理解信息；
- 数据库和文件存储需要备份策略支持恢复。

8.5 部署需求
- 前端：Vite/Nginx 部署；
- 后端：NestJS 单体服务部署；
- 数据层：MySQL 8.0；
- 链层：FISCO BCOS + WeBASE-Front；
- 可使用 Docker Compose 进行本地一键部署。


9 界面要求

9.1 图形要求
- 前端基于 Vue 3 + Element Plus；
- 主布局为“侧边栏 + 顶栏 + 主内容区”；
- 各角色菜单与路由按权限隔离；
- 状态信息（稿件状态、任务状态、链回执）需可视化展示。

9.2 报表与结果展示要求
当前实现要求：
- 版权验证页可展示文件哈希、TxHash、区块高度、存证时间；
- 审稿管理页可展示评分、建议、平均分和裁定结果；
- 区块链管理页可展示节点状态和交易统计。

扩展项（未在当前版本强制实现）：
- 一键导出 PDF 证书；
- 二维码跳转区块链浏览器。

9.3 其他要求
- 关键操作需有加载态与成功/失败反馈；
- 文件上传需校验大小与类型；
- 评分输入限制 0-100；
- 错误消息应友好、可定位。
